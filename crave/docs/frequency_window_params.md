# CRAVE 频率相关参数:不同图像频率需要不同的窗

> **关键**:CRAVE 的 DP 窗参数是在 **3Hz**(stride 10 @30fps)下调出来的。换到更高图像频率(如 30Hz / stride 1)**必须重标定窗**,否则严重抖动。本文档记录标定规则 + 实测依据。

## 为什么频率不同窗就不同

CRAVE value = 在离散 bin 上对"逐帧到 milestone 的距离(emission)"做带转移惩罚的 DP。两类量随频率变化不同:

1. **emission 累积 ∝ fps**:同一段物理时间内,帧数 ∝ fps,逐帧 emission 之和也 ∝ fps。
2. **转移惩罚 `lam·|Δbin|` 按"次"付**:路径净位移一次付一次,与 fps 无关。

→ 频率升高,emission 相对转移惩罚被放大 ∝ fps,DP 更"贴每帧最近 milestone"→ 更抖。**要保持同等平滑度,`lam` 必须 ∝ fps 同步放大。** 时间窗(中值、在线固定滞后)和 proprio 的 Δ 帧距按"秒"对齐,也 ∝ fps。

## 标定规则(通用,任意 fps)

设基准 = 3Hz,目标频率 f(Hz),缩放因子 `k = f / 3`:

| 参数 | 含义 | 缩放 | 3Hz 基准 |
|---|---|---|---|
| `lam`(转移惩罚) | DP 平滑/单调刚度 | **× k**(随 emission 累积率) | 8 |
| 中值窗 `medw` | 后处理平滑(帧) | **× k**(同秒数) | 9(3s) |
| 在线固定滞后 `L` | 因果解码延迟(帧) | **× k**(同秒数) | 12(4s) |
| proprio Δ 帧距 `dt` | state 差分的时间跨度 | **× k**(同 Δt) | 1(1/3s) |
| 端点门控 `tn` | start/end 锚的时间门 | 用归一时间, 与 fps 无关 | tn<0.3 / >0.6 |

**常用频率落表:**

| 频率 | stride | `lam` | `medw` | 在线 `L` | proprio `dt` |
|---|---|---|---|---|---|
| **3Hz** | 10 | **8** | **9**(3s) | **12**(4s) | **1** |
| **30Hz** | 1 | **80** | **45**(1.5s) | **90**(3s) | **10** |

> 注:30Hz 的 `medw` 取 1.5s(45)而非 3s——`lam` 放大后阶梯更稳,中值窗可适当缩短减少滞后;`L` 取 3s(90)折中延迟与平滑。这些是经验甜点,非硬性 ×k。

## 实测依据(ep2047 @30Hz,kai0_base)

| 配置 | 帧间抖动 jitter | corr(在线,离线) | 末值 |
|---|---|---|---|
| **标定窗**(lam80/L90/med45) | **0.0005** | **0.894** | 1.00 |
| 未标定(直接用 3Hz 的 lam8/L12/med9) | **0.0041** | 0.762 | — |

→ **窗标定让抖动降 8×、在线≈离线**。直接拿 3Hz 参数套 30Hz 会狂抖(图 `crave_ep2047_final_offline_vs_online.png` 橙色虚线)。这也说明:之前观察到的"30Hz 噪声"主因是**窗没随频率调**,臂噪声只是次要(~40%,见 [METHOD 臂噪声讨论])。

## 挖矿频率 vs 输出频率(实测:挖矿频率几乎不影响)

同一批 82 个 kai0_base episode,对比 **[30Hz挖矿+30Hz输出]** vs **[3Hz挖矿+3Hz输出]**,最终都在 ep2047 上算 value:

| | 3Hz 挖+3Hz 出 | 30Hz 挖+30Hz 出 |
|---|---|---|
| milestone 数 | **20** | **20**(相同) |
| milestone P_k 分布 | 0.01…0.6…0.85…0.99 | 0.0…0.6…0.84…0.96(**几乎一致**) |
| ep2047 value 点数 | 263 | 2629(10×分辨率) |
| 末值 | 1.00 | 1.00 |
| **corr-with-AE** | **0.80** | **0.78**(基本一样) |
| 两者互相关(对齐时间) | — | 0.62(共享宏观形状) |

**2×2 解耦实验(挖矿频率 × 输出频率,ep2047,`crave_freq_3way.py`)** 完整分离两个因素:

| 配置 | milestone | 末值 | corr-with-AE |
|---|---|---|---|
| ① 3Hz挖 + 3Hz出 | 20 | 1.00 | **0.80** |
| ② 30Hz挖 + 3Hz出 | 20 | 0.95 | **0.77** |
| ③ 3Hz挖 + 30Hz出 | 20 | 1.00 | **0.80** |
| ④ 30Hz挖 + 30Hz出 | 20 | 1.00 | **0.78** |

只变挖矿频率:**corr(①,②)=0.61**(3Hz 输出)/ **corr(③,④)=0.61**(30Hz 输出)。

- **挖矿频率:不改 value 质量,但稳定移动具体曲线**——①↔② 与 ③↔④ 互相关都 **~0.61**(不论 3Hz 还是 30Hz 输出都一致),而 milestone 数(20)、corr-with-AE(0.77–0.80)几乎不变。原因:30Hz 挖矿时 KMeans 看 10× 冗余帧,中心被偏向"时间稠密/停留久"的相位,milestone 落位略移 → 同样好但不同的曲线。**所以"挖矿频率无关"要修正为:不影响质量,但稳定改变曲线形状(corr~0.61)**。
- **输出频率 = 分辨率**:①↔③、②↔④(同挖矿)宏观一致,30Hz 输出只是更细(2629 vs 263 点)。

→ **实用建议**:**挖矿用 3Hz 即可**(质量与 30Hz 同、10× 便宜;30Hz 挖矿是不同但不更好的曲线);**输出按需选分辨率**(3Hz 省 / 30Hz 细),把 3Hz 挖出的模型套到 30Hz 特征即可。视频 `crave_freq_4way_ep2047.mp4`(2×2 四配置同步)/ 图 `crave_mine_freq_compare.png`。

## 离线 vs 在线读出(与频率正交)
- **离线**(全局 Viterbi:前向 + 末端奖励 + 反向 backtrace):非因果,最优,用于离线打标。
- **在线**(固定滞后 Viterbi,延迟 L 帧):因果,零训练,corr 0.89 ≈ 离线,用于实时监控。详见 [METHOD §3.6 / roadmap]。

脚本:`crave_ep2047_final.py`(离线 vs 在线 @30Hz)· `crave_causal_dp_test.py`(固定滞后验证)· `crave_30hz_test.py`(30Hz vs 3Hz)。
